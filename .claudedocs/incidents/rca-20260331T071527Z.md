Root cause: `web/run_direct.sh` and `run_dashboard.sh` trusted any successful response from `http://localhost:$PORT/health` during startup.

When an old or foreign backend was still answering on `:8000`, startup could:

- report the backend as healthy,
- continue to start the frontend or load config,
- then discover that the newly launched backend had already died or never owned the port.

Contributing factors:

- startup only checked health, not listener ownership,
- restart flow checked ports immediately after stop without waiting for release,
- wrapper startup (`run_dashboard.sh`) did not verify that `.running_pids` matched the live listener.

Fix implemented:

- added shared helpers in `scripts/common_env.sh` to wait for port release and inspect the listener PID,
- updated `web/run_direct.sh` to require that the launched backend PID owns the backend port before declaring success,
- updated `run_dashboard.sh` to wait for `.running_pids`, confirm the backend PID is alive, and confirm that same PID owns the listener before proceeding,
- added a regression test covering the false-positive health-listener scenario.

Validation:

- `bash -n run_dashboard.sh web/run_direct.sh web/stop_direct.sh stop_dashboard.sh scripts/common_env.sh`
- `./.venv/bin/python -m pytest tests/test_run_direct_startup_guard.py -q`

Notes:

- Additional existing test failures in the repo are unrelated to this change and stem from current import-path issues in backend test setup.
